Portfolio 1: Reading the EEGLAB Source Code

Cynthia Deng

2026-03-12

Note: The full eeglab.m file referenced in this portfolio can be found in the p06/scripts folder.

Goal

This portfolio documents my process of reading eeglab.m, the main entry point of the EEGLAB toolbox (version 12, circa 2012). I had used EEGLAB through the GUI before, but had never looked at the source code. The goal was to understand what the software is actually doing at the code level, not just as a user clicking through menus.

I should be honest that I worked through much of this with Claude. Claude helped me identify what each section was doing, explained unfamiliar syntax, and helped me figure out which parts were worth focusing on. The understanding in this portfolio is my own, but the process of getting there was genuinely collaborative.

What the File Is Doing

eeglab.m is the application launcher, the entire file is roughly 2000 lines. When a researcher types eeglab at the MATLAB prompt, this is the code that runs. Its responsibilities are: checking the environment, setting up file paths, building the graphical interface, loading plugins, and then returning control to the user.

It does not contain any EEG signal processing. Filtering, ICA, epoching, and statistics all live in separate files organized across several subfolders. This file’s job is to make all of those accessible through a coherent interface. Recognizing this distinction early made the file much easier to read. I stopped looking for the science and started reading it as a piece of software infrastructure.

The file proceeds through several phases at startup:

  • Environment checks: MATLAB version validation, duplicate installation detection
  • Path setup: adding all EEGLAB subfolders to MATLAB’s search path
  • Callback string assembly: pre-building the action sequences attached to menu items
  • Menu construction: building the entire GUI menu bar programmatically
  • Plugin loading: scanning for installed plugins and registering them
  • updatemenu(): a large function that keeps the interface synchronized with the current data state

Environment Checks

The first executable code checks whether MATLAB is new enough to support EEGLAB:

vers = version;
if num2str(vers(1)) < 7
    disp('Your Matlab version is too old to run EEGLAB');
    return;
end;

This is a guard clause: check the precondition at the top, exit early with a clear message if it fails. The file does something similar for duplicate EEGLAB installations. If two versions of EEGLAB exist on MATLAB’s path, function calls become unpredictable, and the code warns the user explicitly rather than failing silently later. Both checks reflect practical experience with the kinds of problems real users run into.

Path Setup

MATLAB requires that any function you want to call be on its search path. EEGLAB’s functions are distributed across many subfolders, none of which are automatically accessible. This section adds all of them using a helper function defined later in the same file:

myaddpath( eeglabpath, 'eeg_checkset.m', 'functions/adminfunc' );
myaddpath( eeglabpath, 'supergui.m',     'functions/guifunc'   );
myaddpath( eeglabpath, 'pop_loadbci.m',  'functions/popfunc'   );

The logic is: if the named function is not already accessible, add the specified folder to the path. This is preferable to blindly appending folders on every startup, which would accumulate duplicate entries over repeated calls.

One detail worth noting is how the code handles MATLAB’s Signal Processing Toolbox, which is a paid add-on:

if ~license('test','signal_toolbox')
    addpath( signalpath );  % use EEGLAB's bundled fallback functions
else
    rmpathifpresent( signalpath );  % use the official toolbox instead
end;

EEGLAB ships with its own versions of the signal functions it needs. If the user does not have the official toolbox, the fallback is loaded silently. This kind of adaptation to the user’s environment is good design. It removes a potential barrier to entry that many researchers would not know how to troubleshoot.

Callback Strings

This section took the most effort to understand, and I needed Claude’s help to work through it. Before the menu is constructed, the code builds a set of variables with names like cb_runica, cb_epoch, and cb_reref. They look like ordinary variable assignments, but their contents are strings of MATLAB code:

cb_runica = [ check '[EEG LASTCOM] = pop_runica(EEG);' e_store ]

These are executable strings that will be attached to menu items as callbacks. When the user clicks “Run ICA,” MATLAB evaluates the string stored in cb_runica. The variables check and e_store are also strings, concatenated in to wrap every action with precondition verification and history logging automatically.

For reference, check expands to something like:

'[EEG LASTCOM] = eeg_checkset(EEG, ''data''); ...'

This confirms that data is loaded before attempting to run ICA. The e_store piece logs the command and saves the result. So every single menu action in EEGLAB is actually a three-part sequence, assembled from reusable string fragments before the interface is even built.

I had not seen this pattern before. In Python or R you would pass a function reference as a callback. MATLAB’s uimenu system expects a string, so the authors adapted to that constraint by pre-assembling composite action strings. The double-escaped quotes inside the strings ('') made the code visually noisy, which is part of why this section was difficult to parse initially.

Menu Construction

With the callback strings assembled, the code builds the menu bar. There is no visual interface builder involved. Every item is created explicitly:

file_m  = uimenu( W_MAIN, 'Label', 'File' );
tools_m = uimenu( W_MAIN, 'Label', 'Tools' );

uimenu( tools_m, 'Label', 'Run ICA',        'CallBack', cb_runica, 'foregroundcolor', 'b' );
uimenu( tools_m, 'Label', 'Extract epochs', 'CallBack', cb_epoch );
uimenu( tools_m, 'Label', 'Re-reference',   'CallBack', cb_reref );

Each top-level menu is stored as a handle variable, and submenus are built by passing that handle as the parent. The hierarchy of the code directly reflects the hierarchy visible in the interface. There are hundreds of these calls spanning several hundred lines.

Each menu item also carries a userdata string that encodes when it should be active:

onepoch = 'startup:off;continuous:off';

uimenu( tools_m, 'Label', 'Extract epochs', 'userdata', onepoch, 'CallBack', cb_epoch );

The tag 'startup:off;continuous:off' means this item is disabled at startup and when data is continuous. It becomes enabled only once the data has been epoched. These tags are not visible to the user. They are metadata attached to each menu item describing its own activation condition, which a separate function reads and applies.

updatemenu(): Keeping the Interface in Sync

The largest function in the file, at roughly 1,000 lines, is updatemenu(). It runs after every significant user action to refresh the interface based on the current state of the data.

The function first determines what state the data is in by assembling a list of status tags:

menustatus = {};

% no data loaded:
menustatus = { 'startup' };

% one continuous dataset:
menustatus = { 'continuous_dataset' };

% data has been epoched:
menustatus = { 'continuous_dataset', 'epoched_dataset' };

It also appends secondary condition tags:

if isempty(EEG.icaweights), menustatus{end+1} = 'ica_absent'; end
if isempty(EEG.chanlocs),   menustatus{end+1} = 'chanloc_absent'; end
if ~isempty(ALLERP),        menustatus{end+1} = 'erp_dataset'; end

With the status list built, the function retrieves all menu handles and their userdata strings, then checks for matches:

allmenus = findobj( W_MAIN, 'type', 'uimenu' );
allstrs  = get(allmenus, 'userdata');

indmatch = cellfun(@(x)(~isempty(findstr(num2str(x), 'startup:off'))), allstrs);
set(allmenus(indmatch), 'enable', 'off');

Rather than maintaining an explicit list of which items to enable or disable in each state, the function handles all menu items at once through a single pattern-matching pass over their tags. Adding a new menu item in the future only requires putting the correct tag on it during construction. No changes to updatemenu() are needed.

Reading this function gave me the clearest picture of how EEGLAB models the EEG analysis workflow. The pipeline it assumes is:

  1. Load raw continuous data
  2. Cut the data into epochs around stimulus events
  3. Run ICA to decompose the signal into independent components
  4. Compute ERPs by averaging across trials

Each stage unlocks the next set of tools in the interface. The grayed-out menus are not cosmetic. They reflect a prerequisite structure encoded in the software: you cannot reject epochs before you have epochs, and you cannot inspect ICA components before running ICA. The interface enforces this through the tag system rather than through error messages after the fact.

What Remained Unclear

There is a full section of the file dedicated to global variables that I did not cover in this portfolio. The code declares and initializes a set of variables shared across the entire toolbox, including EEG, ALLEEG, CURRENTSET, and others. Claude explained the general concept to me, that these variables hold the current dataset and session state and are accessible by any function without being passed explicitly. But the deeper implications of how global state is managed across a session, and why certain variables are structured the way they are, was a level of MATLAB-specific design I did not feel confident enough to write about meaningfully. I flagged it as something to return to.

The history and logging system was also confusingn. There is a function called eegh() that appears constantly throughout the code, and a global variable ALLCOM that seems to accumulate something over the course of a session. I have a rough sense that this is related to reproducibility, that EEGLAB is keeping some kind of record of what the user does.

I followed the plugin loading section only at a conceptual level, but there were details I skipped. EEGLAB scans its plugins/ folder for files named eegplugin_*.m and calls each one at startup. The general idea of a registration system is clear enough. What happens beyond that, how plugins interact with each other, what the error handling looks like in practice, I did not examine closely enough to say anything reliable.
